# 项目简历与面试文档生成提示词模板

> 用途：把任意代码项目整理成“简历项目经历 + 面试自我介绍 + 高频面试题回答 + Markdown 文档”。  
> 使用方式：在新的项目目录下，把下面的提示词复制给 AI / Codex / ChatGPT，让它先阅读项目，再生成一份可复用的面试准备文档。  
> 适合场景：课程项目、开源项目、AI 生成项目、实习项目、GitHub 展示项目、个人作品集项目。

---

## 一键复制版

```text
请你扮演“项目简历包装与面试教练”，帮我把当前代码项目整理成可以写进简历、并能应对面试追问的 Markdown 文档。

要求：
1. 先不要改代码，先阅读项目结构、README、依赖文件、配置文件、前后端入口、核心业务逻辑、测试文件和部署文件。
2. 先判断这个项目是什么类型：前端项目、后端项目、全栈项目、AI 应用项目、数据分析项目、工具类项目，或混合项目。
3. 基于真实代码内容提炼项目亮点，不要凭空编造生产经验、性能指标、用户规模或商业落地结果。
4. 按简历可用的表达方式整理项目经历，不要机械地写 S/T/A/R 四个字母，而是把 STAR 思路自然融合到项目 bullet 里。
5. 输出不同版本的简历写法：推荐版、精简版、偏后端版、偏前端版、偏 AI/算法版、偏全栈版。如果某个方向和项目不匹配，可以说明并跳过。
6. 准备面试时可能被问到的问题，并给出可直接复述的回答。回答要围绕：项目背景、技术架构、核心流程、技术难点、我的贡献、测试验证、部署方式、项目边界和后续优化。
7. 如果项目和 AI 应用相关，必须单独检查：Agent 流程、工具调用、上下文注入、RAG/知识库、回答依据、执行链路、评测集、安全边界和日志审计。
8. 面试回答要自然、具体、能讲清楚“为什么这么做、怎么实现、带来什么价值”，不要只堆技术名词。
9. 明确哪些内容适合写进简历，哪些内容面试时可以讲，哪些内容不要夸大。
10. 最后生成一个 Markdown 文件，文件名建议为：`项目简历与面试题整理.md`。

请生成的 Markdown 至少包含以下部分：
- 项目一句话总结
- 简历项目经历：推荐版、精简版、不同岗位方向版
- 30 秒、1 分钟、3 分钟项目介绍
- 项目架构讲解
- 核心业务流程
- 技术亮点
- AI 应用亮点：可观测、可溯源、可评测、可增强
- 面试高频问题与参考回答
- 面试官可能深挖的代码点
- 简历措辞避坑
- 项目不足与后续优化方向
- 最推荐背下来的核心回答

输出风格：
- 使用中文。
- 适合求职和面试，不要像课程笔记。
- 语言专业但不要夸张。
- 能直接复制到简历或面试稿中。
- 如果项目当前只是 Demo / 原型 / 课程实践，请用稳妥口径表达，不要包装成大规模生产项目。
```

## 带变量的通用版

把下面的变量改成你的真实情况，再复制给 AI。

```text
你是我的“项目简历包装与面试教练”。请基于当前项目代码，帮我生成一份可复用的 Markdown 文档，用于写简历和准备面试。

我的目标岗位：
【填写：后端开发 / 前端开发 / 全栈开发 / AI 应用开发 / 数据分析 / 算法工程 / Java 开发 / Python 开发 / 实习生等】

我的经验阶段：
【填写：应届生 / 实习生 / 1 年经验 / 2-3 年经验 / 转行求职 / 课程项目包装等】

项目定位：
【填写：课程项目 / 个人项目 / 开源项目 / 公司实习项目 / AI 生成后继续完善的项目 / GitHub 展示项目】

我希望突出：
【填写：工程能力 / 后端接口 / 前端页面 / AI Agent / 数据处理 / 数据库设计 / 安全治理 / 测试 / 部署 / 性能优化等】

请你执行以下流程：

第一步：项目阅读
- 阅读项目根目录结构。
- 阅读 README / docs。
- 阅读依赖文件，例如 package.json、requirements.txt、pyproject.toml、pom.xml、go.mod 等。
- 阅读配置和部署文件，例如 Dockerfile、docker-compose.yml、.env.example、CI 文件等。
- 找出前端入口、后端入口、核心业务逻辑、数据库访问、测试文件。
- 如果是 AI 项目，找出模型调用、Agent 流程、工具调用、RAG、MCP、工作流编排、trace/evidence、评测集、安全校验和日志审计等位置。

第二步：项目理解
- 用一句话说明项目解决什么问题。
- 说明用户是谁。
- 说明项目的核心业务流程。
- 说明主要技术栈分别负责什么。
- 说明项目中最适合写进简历的亮点。
- 如果是 AI 应用项目，用“四个可”整理亮点：可观测、可溯源、可评测、可增强。
- 说明哪些点不能夸大。

第三步：简历包装
- 生成“推荐版”项目经历。
- 生成“精简版”项目经历。
- 根据目标岗位生成定制版项目经历。
- 每条 bullet 都要体现：背景/问题、行动/实现、结果/价值。
- 不要机械写 STAR 四个标题，要写成自然的简历项目描述。
- 不要编造数字指标，除非代码或文档中能证明。

第四步：面试准备
- 生成 30 秒、1 分钟、3 分钟项目介绍。
- 生成项目架构讲法。
- 生成核心流程讲法。
- 生成 20 个以上高频面试题和参考回答。
- 回答要能直接说出口，不要写成论文。
- 对关键技术点准备追问，例如为什么用这个技术、怎么实现、有什么风险、如何优化。
- 如果是 AI 应用项目，额外生成关于 Agent 黑盒、回答依据、工具安全、RAG 边界、Text-to-SQL 评测、幻觉控制、成本和延迟的追问。

第五步：风险与边界
- 说明项目当前是 Demo、原型、学习项目还是生产项目。
- 说明生产化还差什么。
- 说明简历里哪些说法稳妥，哪些说法容易被追问或显得夸大。

第六步：生成 Markdown
- 在项目根目录生成一份 Markdown 文档。
- 文件名建议：`项目简历与面试题整理.md`。
- 文档结构清晰，适合后续反复修改和背诵。
```

## 更强约束版，防止 AI 乱吹

如果你担心 AI 把项目包装得太夸张，用这个版本。

```text
请帮我把当前代码项目整理成简历项目经历和面试准备文档，但必须严格遵守以下限制：

1. 只能基于项目真实文件、README、源码、测试和配置来总结。
2. 不允许编造用户规模、QPS、性能提升百分比、企业上线、生产落地、营收、并发量等无法证明的信息。
3. 如果项目是 Demo、课程项目、原型项目或本地演示项目，必须用稳妥口径表达。
4. 简历 bullet 要体现项目价值，但不要过度包装。
5. 面试回答要能经得起追问，必须能对应到项目中的具体文件或模块。
6. 如果某个技术点只是预留入口，没有完整实现，要明确说“预留”或“可扩展”，不要说成已完整落地。
7. 如果项目存在不足，比如权限体系不完整、测试覆盖不全面、未做生产部署，要在“项目边界和后续优化”里诚实说明。
8. 如果是 AI 应用项目，不能把规则层评测说成真实模型准确率，不能把关键词检索说成完整向量 RAG，不能把日志说成完整生产级 tracing。

请输出一份 Markdown 文档，内容包括：
- 项目真实定位
- 简历推荐写法
- 精简写法
- 不同岗位版本
- 面试自我介绍
- 架构说明
- 核心流程
- 技术亮点
- AI 应用专项亮点
- 高频面试题与回答
- 可能被深挖的代码点
- 项目不足与优化方向
- 简历措辞避坑
```

## 简历 bullet 生成规则

好的项目 bullet 不只是“我用了什么技术”，而是要同时包含：

- **业务场景**：这个功能解决什么问题。
- **技术动作**：你用什么方式实现。
- **工程价值**：带来什么效果。

推荐句式：

```text
- 面向【业务场景】，基于【技术栈】实现【核心功能】，支持【具体能力】，提升【价值】。
- 设计【机制/模块】，将【数据/流程/权限/任务】进行【处理方式】，解决【问题】。
- 基于【框架/工具】封装【接口/服务/流程】，支撑【前端页面/业务流程/智能体调用】。
- 实现【安全/测试/部署/监控】能力，增强系统的【稳定性/可维护性/可追溯性/可扩展性】。
```

不推荐句式：

```text
- 熟练使用 React、FastAPI、MySQL、Docker。
- 负责项目开发。
- 完成前后端功能。
- 提升效率 90%。
- 实现企业级高并发系统。
```

原因：

- 太空泛。
- 没有体现项目价值。
- 没有说明你具体做了什么。
- 容易被面试官追问后露馅。

## 面试回答生成规则

每个回答建议按这个顺序组织：

1. **先说背景**：为什么要做这个功能。
2. **再说方案**：你怎么设计。
3. **然后说实现**：用到了哪些文件、模块、技术。
4. **最后说价值**：解决了什么问题，还有什么不足。

模板：

```text
这个功能主要解决【问题】。
我的方案是【方案】。
具体实现上，我在【模块/文件/服务】中做了【关键动作】。
这样做的价值是【效果】。
目前它还有【不足】，如果继续优化我会【后续方向】。
```

示例：

```text
这个功能主要解决不同业务数据容易混用的问题。
我的方案是设计 workspace 业务空间，把上传数据、字段画像、报告摘要和数据库表绑定起来。
具体实现上，用户提问时后端会注入 workspace 上下文，并限制 Agent 只能查询当前空间授权的表和字段。
这样可以减少跨表误查，也让 AI 查询过程更可控、可追踪。
```

## 适配不同岗位的突出重点

### 后端开发

重点突出：

- API 设计。
- 数据库访问。
- 业务服务拆分。
- 数据安全。
- 权限校验。
- 测试。
- 部署。

少写：

- 页面审美。
- 视觉细节。
- 太多 AI 名词堆叠。

### 前端开发

重点突出：

- 页面结构。
- 组件拆分。
- 状态管理。
- 接口封装。
- 图表展示。
- 错误处理。
- 用户体验。

少写：

- 过深的数据库细节。
- 不熟悉的后端安全实现。

### AI 应用开发

重点突出：

- 大模型调用。
- Agent 流程。
- 工具调用。
- RAG / MCP / function calling。
- 上下文注入。
- Agent trace / 工具调用过程。
- evidence / 回答依据 / 数据来源。
- Text-to-SQL 或模型效果评测。
- 工具安全、SQL 安全、权限边界。
- 幻觉控制。
- 安全边界。

少写：

- 只写页面和 CRUD。

### 全栈开发

重点突出：

- 前后端完整链路。
- 接口联调。
- 数据流。
- 部署。
- 测试。
- 端到端业务闭环。

少写：

- 单点技术过深但不能自圆其说。

### 数据分析 / BI

重点突出：

- 数据导入。
- 数据清洗。
- 指标体系。
- 报告生成。
- 图表分析。
- 业务洞察。

少写：

- 和业务分析无关的底层框架细节。

## 检查清单

生成文档后，可以用这份清单检查质量：

- [ ] 是否说明项目真实定位？
- [ ] 是否有一句话总结？
- [ ] 是否有简历可直接粘贴版本？
- [ ] 是否有精简版？
- [ ] 是否按目标岗位做了取舍？
- [ ] 是否避免了编造指标？
- [ ] 是否讲清楚技术栈分别负责什么？
- [ ] 是否讲清楚用户操作后的完整链路？
- [ ] 是否准备了面试高频问题？
- [ ] 是否准备了被追问时的回答？
- [ ] 是否写了项目不足和后续优化？
- [ ] 是否有最推荐背下来的核心回答？

## 最终交付要求

当你让 AI 使用这个模板时，可以在最后加一句：

```text
请直接在当前项目根目录生成 Markdown 文件，并在回答中告诉我文件路径。不要只在聊天窗口输出一大段内容。
```

如果你只想先看内容，不想写文件，可以改成：

```text
先不要写文件，请先在聊天中输出 Markdown 草稿，我确认后再生成文件。
```
