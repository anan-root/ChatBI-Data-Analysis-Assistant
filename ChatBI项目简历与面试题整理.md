项目里我重点设计了 workspace 业务空间。每份上传数据会生成一个业务空间，里面保存原始文件信息、业务类型、字段画像、数据质量、报告摘要和对应数据库表。用户在某个空间下提问时，后端会把这些上下文注入给 Agent，并限制它只能查询当前空间允许的表和字段。为了进一步保证安全和可信，我加了只读 SQL 校验、禁止 `SELECT *`、表字段白名单、审计日志、Agent trace 和 evidence。前端可以展开看到意图分流、工具调用、SQL 是否被拦截、使用了哪些字段和命中了哪些指标口径。

最终项目形成了“数据上传 -> 自动清洗 -> 业务空间生成 -> 自然语言问数 -> 工具调用可观测 -> 回答依据可追溯 -> 规则评测可回归”的 AI BI 原型闭环，适合用作 AI 应用开发岗位的项目展示。

## 4. 项目架构怎么讲

### 4.1 按层讲

可以把项目分成五层：

1. **前端交互层**  
   使用 `React + Vite`，负责页面展示和用户操作，包括聊天框、文件上传、业务空间、BI 看板、报告和审计日志。

2. **后端 API 层**  
   使用 `FastAPI`，负责接收前端请求，提供聊天、上传、清洗、业务空间、报告、审计等接口。

3. **数据处理层**  
   使用 `pandas` 处理 CSV/Excel，包括读取文件、清洗字段、识别字段类型、生成预览数据和导入任务。

4. **智能体编排层**  
   使用 `LangGraph` 管理 Agent 执行流程，判断用户问题应该走普通回答、SQL 查询、Python 计算还是图表分析。

5. **工具与存储层**  
   使用 `MCP` 暴露数据库查询、Python 执行、业务空间查询等工具；使用 `PostgreSQL` 存储业务数据；使用文件或数据库保存元数据和审计日志。

### 4.2 一句话架构

前端负责交互，FastAPI 负责接口，pandas 负责清洗，LangGraph 负责决策，MCP 负责工具调用，PostgreSQL 和文件系统负责数据落地。

### 4.3 请求链路

用户输入问题后的链路：

1. 用户在 React 前端聊天框输入问题。
2. 前端把 `message`、`history` 和可选的 `workspace_id` 发送给 FastAPI。
3. FastAPI 判断是否存在 workspace 上下文。
4. 如果有 workspace，则构造系统提示，加入业务类型、字段画像、报告摘要和 SQL 访问限制。
5. 请求进入 LangGraph 图流程。
6. Agent 判断问题类型：普通咨询、Python 计算、业务数据查询或图表分析。
7. 如果需要查数，就通过 MCP 调用数据库或业务空间工具。
8. SQL 执行前经过只读校验、表白名单和字段白名单。
9. 后端把最终回答返回给前端。
10. 前端展示回答内容。

## 5. 高频面试题与回答

### 5.1 这个项目是做什么的？

**回答：**

这个项目是一个 ChatBI 数据分析助手，面向企业经营分析场景。它支持用户上传 Excel 或 CSV 数据，系统自动完成清洗、字段识别和业务空间生成。之后用户可以通过自然语言提问，比如查询销售额、分析趋势、生成图表或输出诊断报告。

它的核心不是普通聊天，而是把数据上传、清洗、问数、分析、报告和审计串成一个闭环，让非技术业务人员也能完成基础数据分析。

**可以补一句：**

我在项目里重点做了业务空间隔离和 SQL 安全治理，防止 Agent 查错表、查错字段或越权访问数据。

---

### 5.2 它和普通 ChatGPT 问答有什么区别？

**回答：**

普通 ChatGPT 主要是通用问答，它不知道企业本地数据，也没有严格的数据访问边界。如果直接问它业务数据问题，它很容易凭经验回答，甚至产生幻觉。

这个项目是面向企业数据分析的。用户先上传真实数据，系统清洗后生成业务空间。后续用户提问时，后端会把当前 workspace 的字段、业务类型、报告摘要和访问限制注入给 Agent。Agent 不是凭空回答，而是可以调用 SQL、Python、图表分析等工具完成分析。

同时，查询前会经过只读 SQL、表字段白名单和审计日志机制，保证数据查询过程更安全、更可追踪。

---

### 5.3 为什么要做业务空间 workspace？

**回答：**

业务空间解决的是“不同业务数据不能混用”的问题。

比如用户上传了销售数据、库存数据和用户增长数据，如果没有空间隔离，Agent 在回答问题时可能会查错表，或者把不同业务口径混在一起分析。这样结果就不可信。

所以我设计了 workspace 机制。每份上传数据会生成或归入一个业务空间，空间里保存文件名、业务类型、字段画像、数据质量、报告摘要和对应的数据表。用户在某个 workspace 下提问时，后端会把这个上下文传给 Agent，并限制它只能查询当前 workspace 授权的数据。

**重点句：**

workspace 的价值是让 Agent 的回答从“泛泛问答”变成“基于当前业务数据的受控分析”。

---

### 5.4 用户上传 CSV 或 Excel 后，系统做了什么？

**回答：**

上传后主要做五步：

1. 读取文件，支持 `CSV`、`XLSX` 和 `XLS`。
2. 清洗数据，包括删除全空行列、处理重复列名、去重、去掉字符串空格、填充空值。
3. 类型推断，比如识别日期字段、数值字段。
4. 字段角色识别，把字段分成指标、维度、时间和 ID 等类型。
5. 生成导入任务、字段画像、预览数据和建议表名，用户确认后可以入库形成业务空间。

这里的重点是，系统不会直接把脏数据丢给 Agent，而是先把数据结构化，让后续问数和报告生成更可靠。

---

### 5.5 LangGraph 在项目里起什么作用？

**回答：**

`LangGraph` 在项目里主要负责 Agent 流程编排。

用户的问题不是都走同一条路。有的问题是普通咨询，有的问题需要查 SQL，有的问题需要跑 Python 计算，有的问题需要图表分析。LangGraph 可以把这些步骤组织成一个图流程：先判断问题意图，再根据意图进入不同节点，比如业务分流、获取表结构、数据分析 Agent、Python 工具执行等。

它的价值是让 Agent 的执行过程更可控，而不是一次性把所有任务都交给大模型自由发挥。

---

### 5.6 MCP 是什么？为什么用 MCP？

**回答：**

MCP 可以理解为给 Agent 使用外部工具的一套协议或接口规范。

在这个项目里，我把数据库查询、Python 执行、机器学习分析、业务空间查询等能力，通过 MCP server 暴露成工具。LangGraph Agent 在分析问题时，可以按需调用这些工具。

使用 MCP 的好处是工具边界比较清楚，每个工具服务可以独立维护。后续如果要扩展新的分析工具，不需要把所有逻辑都堆在一个大函数里，只要新增 MCP 工具并接入 Agent 流程即可。

**简短版：**

LangGraph 决定流程怎么走，MCP 提供 Agent 可以调用的工具。

---

### 5.7 Agent 如何判断该查 SQL、跑 Python，还是直接回答？

**回答：**

项目里设计了一个意图分流流程。用户问题进入后，Agent 会先判断问题类型：

- 如果是普通咨询，可以直接回答。
- 如果是计算、绘图或代码类任务，可以走 Python 工具。
- 如果是业务数据查询分析，会先获取数据表结构，再生成 SQL 或调用数据分析工具。

这个分流逻辑由 LangGraph 编排，不同类型的问题会进入不同节点。这样可以避免所有问题都走同一套处理逻辑。

---

### 5.8 怎么防止大模型生成危险 SQL？

**回答：**

我做了多层限制。

第一，只允许只读 SQL，比如 `SELECT`、`WITH`、`EXPLAIN`，禁止 `INSERT`、`UPDATE`、`DELETE`、`DROP` 等写操作。  
第二，在 workspace 模式下，只允许查询当前业务空间绑定的数据表。  
第三，对字段也做白名单限制，只允许查询字段画像中存在的字段，并禁止 `SELECT *`。  
第四，每次 SQL 放行或拦截都会写审计日志，便于排查和追踪。

所以项目不是完全信任大模型生成的 SQL，而是在执行前加了一层安全校验网关。

---

### 5.9 什么是只读 SQL？

**回答：**

只读 SQL 指的是只能读取数据，不能修改数据。

在这个项目里，允许的 SQL 主要是 `SELECT`、`WITH`、`EXPLAIN` 这类查询语句。像 `INSERT`、`UPDATE`、`DELETE`、`DROP`、`ALTER` 这类会修改数据库结构或数据的语句都应该被拦截。

这样做是为了防止大模型生成危险 SQL，避免误删、误改或破坏业务数据。

---

### 5.10 表白名单和字段白名单怎么做？

**回答：**

表白名单来自当前 workspace 的元数据。每个业务空间会记录它绑定的数据库表名，所以用户在这个空间下提问时，只允许查询这张或这组授权表。

字段白名单来自上传时生成的字段画像。系统会保存当前数据有哪些字段、字段类型、字段角色等信息。SQL 执行前会校验查询里引用的字段是否属于当前 workspace 的允许字段。

这样 Agent 即使生成了 SQL，也不能随便访问其它业务空间、系统表或不存在的字段。

---

### 5.11 审计日志记录什么？有什么价值？

**回答：**

审计日志主要记录 SQL 查询相关信息，比如：

- 查询是否被允许。
- 如果被拦截，原因是什么。
- 查询来源。
- 对应的 workspace 信息。
- 查询语句或查询摘要。
- 时间信息。

它的价值有两个：

第一是安全，可以看到 Agent 有没有尝试访问不该访问的表或字段。  
第二是可追溯，后续如果用户质疑某个回答，可以回头看当时用了什么查询、是否被安全策略放行。

---

### 5.12 前端有哪些主要模块？

**回答：**

前端是一个 React 工作台，主要模块包括：

- 数据问答：用户输入自然语言问题。
- 批量导入：上传 CSV/Excel 并查看清洗结果。
- 业务空间：查看不同上传数据形成的 workspace。
- BI 看板：展示指标、趋势和排行。
- 分析报告：展示系统生成的诊断型报告。
- 分析模板：沉淀常用分析问题。
- 审计日志：查看 SQL 放行、拦截和访问记录。
- RAG-lite 知识库：展示本地指标字典和业务口径条目。
- AI 评测：展示 Text-to-SQL 规则样例、通过情况和失败明细。
- 聊天增强面板：展示 Agent 执行链路和回答依据。

前端并不是只做聊天框，而是围绕 ChatBI 分析流程做了一个工作台。

---

### 5.13 后端 FastAPI 主要有哪些接口？

**回答：**

后端接口大致分几类：

- 聊天接口：接收用户问题、历史记录和 workspace_id，调用 LangGraph Agent。
- BI 数据接口：返回看板、指标、异常分析、报告、用户增长、变现分析等数据。
- 文件导入接口：支持单文件和批量上传，生成清洗任务。
- 导入任务接口：查询导入任务、下载清洗后文件、确认入库。
- 业务空间接口：列出 workspace，查看某个 workspace 的报告。
- 审计接口：查看 SQL 审计日志，支持管理员令牌校验。
- 导出接口：导出报告、指标和看板数据。

---

### 5.14 聊天接口和普通 BI 接口有什么区别？

**回答：**

普通 BI 接口一般是固定接口，前端请求哪个接口，就返回对应模块的数据，比如看板、指标或报告。

聊天接口更复杂。它接收的是自然语言问题，需要把问题、历史记录和 workspace 上下文交给 LangGraph Agent。Agent 再判断是否要调用 SQL、Python、业务空间或图表工具，最后生成回答。

所以普通 BI 接口是确定性的模块数据接口，聊天接口是智能体驱动的动态分析接口。

---

### 5.15 项目里为什么用 pandas？

**回答：**

因为项目需要处理 CSV 和 Excel，pandas 对表格数据处理非常方便。它可以快速完成文件读取、字段处理、空值统计、去重、类型转换、数据预览和简单统计分析。

在这个项目里，pandas 主要用于导入清洗阶段。上传文件后，系统用 pandas 读取数据，再进行字段标准化、空值处理、日期和数值字段识别，最后生成清洗后的 CSV 和元数据。

---

### 5.16 PostgreSQL 在项目里负责什么？

**回答：**

PostgreSQL 主要用于存储业务数据，以及可选地保存元数据和审计日志。

用户上传并清洗数据后，可以确认入库，形成可以被 SQL 查询的业务表。后续用户在 workspace 中提问时，Agent 生成 SQL 查询 PostgreSQL。项目还支持把导入任务元数据和审计日志同步到 PostgreSQL，便于持久化和后续管理。

---

### 5.17 为什么还要保存元数据？

**回答：**

元数据不是业务明细数据，而是描述这份数据的信息，比如原始文件名、字段列表、字段类型、业务类型、清洗报告、导入状态、对应数据库表名等。

保存元数据的好处是：

- 前端可以展示导入历史和业务空间。
- Agent 可以知道当前 workspace 有哪些字段。
- SQL 白名单可以基于字段画像生成。
- 报告和问答可以复用这份上下文。

简单说，元数据让系统知道“这份数据是什么、能怎么用、能查哪些字段”。

---

### 5.18 如果 workspace 还没入库，用户问明细怎么办？

**回答：**

如果当前 workspace 还没有确认入库，系统不应该调用数据库明细查询。因为没有对应的数据库表可以安全查询。

这种情况下，Agent 可以基于已有的字段画像、数据质量、预览信息和报告摘要做解释，但如果用户要查明细数据，应该提示用户先在导入清洗模块确认入库。

这个逻辑也体现在 SQL 策略里：未入库的 workspace 禁止调用数据库查询工具。

---

### 5.19 如何处理脏数据？

**回答：**

项目在导入阶段做了基础清洗：

- 删除全空行和全空列。
- 标准化字段名。
- 处理重复列名。
- 去除字符串首尾空格。
- 将空字符串转为空值。
- 删除重复行。
- 对数值字段空值填充 0。
- 对文本字段空值填充“未知”。
- 尝试识别日期字段和数值字段。
- 将 pandas 中不适合 JSON 返回的值转换成安全格式。

这样可以保证前端预览、后端接口和后续 Agent 分析不会因为常见脏数据直接报错。

### 5.20 这个项目有哪些测试？

**回答：**

项目使用 `pytest` 做后端核心逻辑测试，重点覆盖几类：

- 数据导入清洗，比如字段处理、空值和 JSON 安全转换。
- workspace SQL 范围校验，比如不能查其它表、不能查询未授权字段。
- 安全校验，比如只读 SQL 拦截。
- 审计日志，比如 SQL 放行和拦截是否能记录。
- 元数据持久化，比如导入任务是否能保存和读取。
- 报告和业务空间相关服务。

我重点测的是项目中“不能出错”的边界逻辑，比如数据安全、查询范围和清洗结果。

### 5.21 Docker Compose 做了什么？

**回答：**

Docker Compose 用于一键启动本地演示环境。

它主要启动三个服务：

- `postgres`：PostgreSQL 数据库。
- `backend`：FastAPI 后端和相关 MCP 服务。
- `frontend`：React 构建后的静态页面，通过 Nginx 提供访问。

同时它还配置了数据卷，用于保存数据库数据、导入任务、导出文件、审计日志和图表输出。这样本地演示时环境更容易复现。

### 5.22 如果部署失败，你会怎么排查？

**回答：**

我会按层排查：

1. 先看容器是否启动成功，比如前端、后端、数据库是否都在运行。
2. 再看环境变量，尤其是数据库连接、API Key、CORS、server_url 等配置。
3. 然后看后端日志，确认 FastAPI 是否启动，MCP 服务是否可访问。
4. 再检查数据库连接，确认 PostgreSQL 是否 healthy，用户名、密码、库名是否正确。
5. 最后看前端请求路径，确认 Nginx 或 Vite 代理是否把 `/api` 转发到后端。

排查思路是从基础服务到接口链路，再到业务逻辑。

### 5.24 这个项目最大的难点是什么？

**回答方向 1：业务空间和上下文**

最大的难点是如何让 Agent 基于当前业务数据回答，而不是泛泛回答。  
我的解决方案是设计 workspace 业务空间，把上传文件、字段画像、业务类型、数据质量、报告摘要和数据库表绑定起来。用户提问时，后端把 workspace 上下文注入给 Agent，同时限制查询范围。

**回答方向 2：SQL 安全**

另一个难点是大模型生成 SQL 不可完全信任。  
我的解决方案是在执行前做只读 SQL 校验、表白名单、字段白名单和审计日志，保证 Agent 即使生成 SQL，也不能随意访问其它表或执行危险操作。

### 5.25 如果让你继续优化，你会做什么？

**回答：**

我会优先优化四个方向：

1. **异步导入**  
   大文件上传和清洗改成异步任务，前端显示任务进度。

2. **权限体系**  
   增加用户登录、角色权限和 workspace 级授权。

3. **Agent 效果评估**

4. **生产化安全**  
   增加数据脱敏、查询限流、资源隔离、Python 沙箱和更完整的审计。

如果面试岗位偏 AI，我会优先讲 Agent 可观测性、回答依据、RAG-lite 和评测集；如果偏后端，我会优先讲异步任务、权限和安全。

### 5.26 项目里有没有 RAG？

**回答：**

项目里现在做的是 RAG-lite，不是完整向量库 RAG。  
我先用本地指标字典维护常见业务口径，比如 GMV、销售额、客单价、ROI、转化率、留存率、活跃用户、毛利率等。

用户提问时，后端会根据问题文本和当前 workspace 字段做关键词匹配，取 top 3 条指标口径注入 Agent 上下文，并在 `evidence.knowledge` 里返回给前端。这样回答不仅能查数据，也能解释业务指标含义。

我会特别说明边界：这个版本没有接入 embedding 和向量数据库，所以我不会说它是完整企业 RAG。它的价值是先把业务口径增强链路打通，后续可以替换成向量检索、权限过滤和知识库管理。

---

### 5.27 为什么不用普通函数调用，非要 LangGraph + MCP？

**回答：**

如果只是一个简单工具，普通函数调用也可以。但这个项目里问题类型比较多，可能是普通问答、SQL 查询、Python 计算、图表分析、机器学习或业务空间查询。

`LangGraph` 适合把这些流程组织成可控节点，比如先分流，再查表结构，再调用工具，再汇总结果。  
`MCP` 适合把外部能力封装成工具服务，降低 Agent 和具体工具实现的耦合。

所以这里用 LangGraph + MCP，是为了让流程更清晰，工具边界更明确，后续扩展更方便。

---

### 5.28 你在这个项目里最能体现价值的地方是什么？

**回答：**

我觉得最能体现价值的是把 AI 问答和真实业务数据分析流程结合起来，而不是只做一个聊天界面。

具体包括三点：

第一，做了数据上传和清洗，让系统有真实数据基础。  
第二，设计了 workspace 业务空间，让 Agent 的回答有明确上下文。  
第三，做了 trace、evidence、RAG-lite、SQL 安全和规则评测，让 Agent 工具调用过程可观测、可追溯、可回归。

这三点保证了项目不是“模型随便答”，而是“在受控数据范围内完成分析”。

---

### 5.29 Agent trace 和 evidence 是怎么做的？

**回答：**

我把聊天接口从只返回 `message` 扩展成同时返回 `trace` 和 `evidence`，但保持原来的 `message` 字段不变，保证前端兼容。

`trace` 主要记录执行过程，比如 traceId、意图分类、当前 workspace、工具计划、工具结果、SQL 摘要和拦截原因。实现上，我从 LangGraph 事件里的 `AIMessage.tool_calls` 和 `ToolMessage` 中提取工具名、参数、SQL 和结果摘要。

`evidence` 主要记录回答依据，比如当前业务空间、字段列表、SQL 是否被允许、命中的指标口径。前端聊天消息下面用折叠面板展示“执行链路”和“回答依据”。

这部分的价值是让 Agent 从黑盒变成可解释链路，面试时可以直接展示模型做了什么、用了什么数据、哪里被安全规则拦截。

---

### 5.30 Text-to-SQL 评测做了什么？

**回答：**

我做的是规则层 Text-to-SQL 评测，不是宣称模型 SQL 生成准确率。评测集放在 `tests/fixtures/text2sql_cases.json`，覆盖合法查询、聚合统计、时间趋势、跨 workspace 查询、`SELECT *`、`DELETE / UPDATE / DROP` 和未入库 workspace 查询等场景。

后端通过 `text2sql_eval.py` 读取样例，调用现有的只读 SQL 校验和 workspace SQL 范围校验，输出总用例数、通过数、分类统计和失败 case。前端增加了“AI 评测”入口，可以展示这些结果。

这样做的价值是：即使还没有真实模型评测，也先把 AI 数据查询里最危险、最不能错的规则边界做成可回归测试。

---

### 5.31 这次你解决过什么实际 Bug？

**回答：**

有一个典型问题是 workspace 下查询明明已经进入正确空间，但接口仍然返回失败。排查时我按服务链路看：前端请求、后端聊天接口、LangGraph 节点、MCP 工具、PostgreSQL 表状态。

最后发现有两类问题：第一是本地 Docker PostgreSQL 和 `.env` 密码不一致，导致服务虽然启动但数据库状态不匹配；第二是 workspace scoped SQL wrapper 里调用底层 MCP 工具时用了同步 `invoke`，在异步 Agent 链路里应该用 `await ainvoke`。

我没有绕开安全校验，而是修正了工具调用方式，并重新提交导入任务让 workspace 表真实落库。这个问题比较能体现我对 AI 应用链路的排查方式：不是只看模型输出，而是从前端、API、Agent、工具、数据库逐层定位。

## 7. 面试官可能深挖的代码点

### 7.1 前端聊天链路

可以这样讲：

前端入口在 `main.jsx`。用户点击发送后，会调用 `sendMessage`，这个函数会整理用户输入、历史记录和 workspace_id，然后调用 `client.js` 里的 `sendChatMessage`。  
`client.js` 负责统一封装请求，把数据 POST 到后端聊天接口。后端返回结果后，前端把 assistant 消息追加到聊天列表里。

### 7.2 后端聊天接口

可以这样讲：

后端聊天接口接收 `user_id`、`message`、`history` 和可选的 `workspace_id`。  
如果带了 workspace_id，就先构造 workspace 上下文，并生成系统提示，要求 Agent 基于当前业务空间回答。  
然后后端创建 LangGraph 图流程，通过流式事件获取 AIMessage 和 ToolMessage，同时提取工具调用、SQL、拦截原因和最终回答。最终接口返回 `message`、`trace` 和 `evidence`，其中 `message` 保持兼容，新增字段用于前端展示执行链路和回答依据。

### 7.3 导入清洗逻辑

可以这样讲：

导入清洗核心在 `import_cleaning.py`。它先根据文件后缀读取 CSV 或 Excel，然后复制出 DataFrame 做清洗。清洗包括删除全空行列、字段名标准化、重复列处理、字符串去空格、空值填充、重复行删除、日期和数值类型推断。  
清洗后会生成 schema profile，包括字段名、类型、缺失率、唯一值数量和字段角色。最后保存清洗后的 CSV 和元数据。

### 7.4 SQL 范围校验

可以这样讲：

SQL 范围校验主要做两件事：先判断 SQL 是否只读，再判断它是否符合当前 workspace 的访问范围。  
如果用户在某个 workspace 下提问，系统会拿到该 workspace 允许访问的表和字段。SQL 执行前会抽取查询中的表名和字段名，如果发现访问了其它表、未授权字段或使用 `SELECT *`，就会拒绝执行并记录审计。

### 7.5 Agent trace 和 evidence

可以这样讲：

Agent trace 的核心在 `agent_trace.py`。它会从 `AIMessage.tool_calls` 中提取工具计划，从 `ToolMessage` 中提取工具结果摘要，并根据结果文本判断是否被拦截。  
聊天接口在 LangGraph 流式执行时累计这些 step，最后组装成 `trace`。同时根据 workspace 字段、SQL step 和 RAG-lite 命中内容组装 `evidence`。前端 `ChatMessages.jsx` 负责把这些内容展示成可折叠面板。

### 7.6 RAG-lite 和 AI 评测

可以这样讲：

RAG-lite 的实现比较轻量，核心是 `knowledge_base.py` 加本地 `metric_dictionary.json`。它根据用户问题和字段名做关键词匹配，命中后把指标解释注入上下文，并返回给前端作为依据。

AI 评测的核心是 `text2sql_eval.py` 和 `tests/fixtures/text2sql_cases.json`。它不调用真实 LLM，而是复用 SQL 安全校验和 workspace 范围校验，验证危险 SQL、跨空间表、字段越权和未入库查询是否能被拦截。

- “支持千万级数据”
- “完全解决幻觉”
- “完全防止 SQL 注入”

除非你确实做了压测、生产部署和安全验证，否则不要写得太满。

### 8.3 面试时诚实但不弱

可以说：

这个项目目前更偏原型和实践项目，不是完整生产系统。但我在设计时已经考虑了生产化会遇到的问题，比如 workspace 隔离、SQL 安全、审计日志、Docker 部署和核心测试。后续如果要生产化，我会继续补权限、多租户、异步任务、监控、沙箱和数据脱敏。

这句话很稳，既不吹，也显得你知道工程边界。

还可以补一句更偏 AI 应用的版本：

它现在已经具备 Agent 工具调用、上下文注入、回答依据、规则评测和安全边界这些 AI 应用关键能力，但还没有做完整企业权限、真实模型效果评测、向量化知识库和生产级 tracing。所以我会把它定位成可演示、可扩展的 AI BI 原型，而不是生产系统。

## 10. 最推荐背下来的核心回答

### 10.1 项目核心价值

这个项目的核心价值是把传统 BI 的“上传、清洗、建模、查数、报告”流程和大模型 Agent 结合起来。我的重点不是让模型随便聊天，而是通过 workspace 上下文、MCP 工具调用、RAG-lite 指标口径、SQL 安全校验、Agent trace 和 evidence，让 Agent 在可控的数据范围内完成业务分析，并且能解释它做了什么、依据是什么。

### 10.2 架构核心句

前端负责交互和执行链展示，FastAPI 负责接口，pandas 负责数据清洗，LangGraph 负责决策流程，MCP 负责工具调用，PostgreSQL 和文件系统负责数据落地，本地指标字典负责轻量业务口径增强。

### 10.3 安全核心句

我没有直接信任大模型生成的 SQL，而是在执行前做只读校验、workspace 表范围校验、字段白名单校验，并记录审计日志；同时用固定样例做 Text-to-SQL 规则评测，保证危险查询、跨空间查询和未入库查询这些边界可以回归验证。

### 10.4 workspace 核心句

workspace 的作用是把每份业务数据、字段画像、报告摘要和数据库表绑定在一起。用户在某个空间里提问时，Agent 只能基于这个空间的数据回答，避免跨业务数据混用。

### 10.5 AI 应用核心句

我把这个项目按 AI 应用工程的四个关键词来做：可观测、可溯源、可评测、可增强。  
可观测是 Agent trace；可溯源是 evidence；可评测是 Text-to-SQL 规则评测；可增强是 RAG-lite 指标口径。

## 11. 面试答题模板

遇到任何项目问题，可以按这个结构回答：

1. **先说业务问题**  
   为什么要做这个功能。

2. **再说技术方案**  
   用了什么技术、分了哪些层。

3. **然后说关键实现**  
   具体怎么做，比如上下文注入、白名单校验、工具调用。

4. **最后说结果价值**  
   提升了什么，比如易用性、安全性、可追溯性、可扩展性。

示例：

> 这个功能主要解决业务数据容易混用的问题。我的方案是设计 workspace 业务空间，把上传数据、字段画像、报告摘要和数据库表绑定起来。用户提问时，后端会注入 workspace 上下文，并限制 Agent 只能查询当前空间授权的表和字段。这样可以减少跨表误查，也让回答过程更可控。

AI 应用版示例：

> 这个功能主要解决 Agent 黑盒和回答缺少依据的问题。我的方案是在聊天接口里保留原有 message，同时新增 trace 和 evidence。trace 记录意图分流、工具调用、SQL 校验和拦截原因，evidence 记录 workspace、字段、SQL 和命中的指标口径。这样前端不仅能展示最终回答，还能展示回答是怎么来的。

- 报告版本管理。
- 数据脱敏。
- Python 执行沙箱。

这些方向也可以作为面试时的“后续规划”回答。
