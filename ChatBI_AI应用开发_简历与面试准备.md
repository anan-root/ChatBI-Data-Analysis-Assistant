### 0.7 应届生技术追问题库

> 面试官通常不会只听项目介绍，会顺着技术栈追基础。下面这些问题优先背“短答”，能展开再展开。

#### A. 项目链路

**1. 用户从上传文件到拿到回答，完整链路是什么？**

> 前端上传 CSV/Excel，后端用 pandas 清洗和识别字段，生成 workspace 和字段画像；用户提问时带上 workspaceId，后端注入当前业务空间上下文，LangGraph 判断意图并调用 MCP 工具；SQL 类问题会先做只读、字段白名单和 workspace 范围校验，通过后查询数据库，最后返回 message、trace 和 evidence。

**2. 为什么要有 workspace？不用直接查表行不行？**

> 不建议直接查表。workspace 是数据边界，把上传数据、字段画像、报告摘要和可查询表绑定起来。这样 Agent 回答时知道当前只能查哪些表和字段，避免不同用户或不同业务数据混用。

**3. trace 和 evidence 有什么区别？**

> trace 更偏过程，记录意图识别、工具调用、SQL 校验、工具结果和失败原因；evidence 更偏依据，记录 workspace、字段、SQL 和命中的指标口径。一个回答“怎么来的”，一个回答“依据是什么”。

**4. 这个项目里最难的点是什么？**

> 不是模型回答本身，而是让 Agent 在可控边界内使用真实数据。难点包括 workspace 上下文怎么注入、SQL 怎么限制只读和授权范围、工具调用失败怎么追踪、回答怎么带依据。

#### B. Python / FastAPI

**5. FastAPI 为什么适合这个项目？**

> FastAPI 对异步接口、Pydantic 参数校验和自动接口文档支持比较好，适合做 AI 应用后端。这个项目里聊天、上传、workspace、审计和评测接口都可以用清晰的 request/response 模型约束。

**6. Pydantic 在项目里解决什么问题？**

> 主要解决接口入参和出参结构化校验，比如用户问题、workspaceId、聊天返回的 message/trace/evidence。如果没有结构化模型，前后端字段容易不一致，错误也更晚才暴露。

**7. pandas 清洗 Excel/CSV 时要注意什么？**

> 要注意编码、空值、重复行、列名标准化、类型推断和日期/数字字段识别。对 ChatBI 来说，还要把字段变成可给 Agent 理解的字段画像，比如字段名、类型、样例值和业务角色。

**8. Python 异步里常见坑是什么？**

> 常见坑是同步阻塞函数放进异步链路，或者在 async 流程里调用同步工具导致事件循环卡住。这个项目里 MCP 工具调用也要注意同步和异步方法匹配。

#### C. SQL / 数据库

**9. 你怎么防止大模型生成危险 SQL？**

> 执行前做规则校验：只允许 SELECT 类只读查询，拦截 DELETE、UPDATE、DROP 等危险语句；禁止 SELECT *；校验表名和字段是否属于当前 workspace；未入库的数据禁止明细查询，并记录审计日志。

**10. 为什么禁止 SELECT *？**

> 一是可能暴露不需要的敏感字段，二是返回数据过大，三是不利于回答依据解释。让模型明确选择字段，可以增强可控性和可追溯性。

**11. 字段白名单怎么理解？**

> 当前 workspace 里有哪些字段是已知的，SQL 只能查这些字段。比如只允许查“销售额、月份、地区”，模型如果生成“利润”这种不存在或未授权字段，就应该拦截。

**12. SQL 注入和模型生成危险 SQL 是一回事吗？**

> 不完全一样。SQL 注入通常是用户输入被拼接进 SQL，模型生成危险 SQL 是 LLM 自己产出不安全语句。防护思路有重叠，都要做参数化、只读限制、关键字拦截和权限边界。

**13. 如果 SQL 执行报错，你会怎么处理？**

> 先记录错误和原 SQL，再判断是字段名、表名、语法还是权限问题。生产上可以把安全处理后的错误反馈给模型重试，但不能把数据库敏感错误完整暴露给用户。

#### D. Agent / LangGraph / MCP / RAG

**14. LangGraph 在这里负责什么？**

> 负责 Agent 流程编排，比如先识别问题意图，再决定是否调用 SQL、Python、workspace 等工具，最后汇总回答。它更像工作流控制，不只是一次模型调用。

**15. MCP 在这里负责什么？**

> MCP 把数据库查询、Python 图表、机器学习、业务空间这些外部能力封装成工具服务，让 Agent 通过统一协议调用，降低主流程和工具实现的耦合。

**16. MCP 比 Function Call 多解决了什么？**

> Function Call 适合快速接本地函数；MCP 更强调工具服务化、跨工具统一协议和边界清晰。代价是部署链路更复杂，要保证 MCP 服务、数据库、环境变量和异步调用都正常。

**17. RAG-lite 为什么不用向量库？**

> 当前只是 Demo 和指标口径增强，先用本地 JSON 指标字典做关键词匹配，成本更低、实现更简单。它不能等同完整 RAG，但足够展示“回答前注入业务口径”的思路。

**18. 怎么减少大模型幻觉？**

> 不靠一句提示词解决，而是多层控制：限定 workspace 数据范围，SQL 执行前校验，回答时强制带 evidence，指标口径来自字典，无法查询到的数据不要编造。

**19. 多 Agent 下一版怎么拆？**

> Supervisor 负责拆解任务和调度；SQL Agent 负责生成和校验 SQL；数据处理 Agent 负责清洗、聚合和格式转换；分析 Agent 负责趋势、对比和解释；问答 Agent 负责把结果组织成自然语言并引用 evidence。

#### E. 前端 / React

**20. 前端怎么展示 trace/evidence？**

> assistant message 除了 message 文本，还保存 trace 和 evidence 字段。页面可以用折叠区域展示执行链路、SQL、安全校验状态、workspace、字段和指标口径。

**21. React 里状态怎么设计？**

> 聊天消息可以是数组，每条 message 包含 role、content、trace、evidence、loading/error 等字段；发送问题时先插入 user message 和 loading 状态，接口返回后更新 assistant message。

**22. 前端遇到 502 怎么排查？**

> 先看后端是否启动，再看代理配置和接口路径；然后看 Docker/PostgreSQL/MCP 服务是否正常；最后看后端日志。502 不是前端页面问题，通常是代理后面的服务不可用。

**23. 为什么不把所有逻辑放前端？**

> 数据清洗、SQL 校验、数据库查询和 Agent 工具调用都属于后端职责。前端负责交互和展示，后端负责数据、安全和业务逻辑，这样边界更清楚。

#### F. 测试 / 评测

**24. 73 个测试主要测什么？**

> 主要覆盖数据清洗、workspace 上下文、SQL 安全、Agent trace、知识检索、Text-to-SQL 规则评测、审计日志等核心逻辑。重点不是测模型有多聪明，而是测工程边界是否稳定。

**25. Text-to-SQL 评测为什么不说准确率？**

> 因为当前评测是规则层，不调用真实 LLM，不评估模型生成 SQL 的语义正确率。它主要验证危险 SQL、跨 workspace、字段越权、SELECT * 和未入库查询能不能被拦截。

**26. 单元测试和集成测试有什么区别？**

> 单元测试测单个函数或模块，比如 SQL 校验；集成测试测多个模块一起工作，比如聊天接口能否结合 workspace、Agent 和 evidence 返回完整结果。

#### G. Docker / 部署 / 排查

**27. Docker Compose 在项目里起什么作用？**

#### H. 计算机基础追问

**30. HTTP GET 和 POST 区别？**

> GET 通常用于查询，参数在 URL 上，语义上应幂等；POST 通常用于提交数据或触发处理，请求体可以更复杂。聊天、上传这类操作更适合 POST。

**31. 同步和异步区别？**

> 同步是一个任务执行完再往下走；异步是在等待 I/O 时释放执行权，让服务能处理其他请求。AI 应用里模型调用、工具调用和数据库访问都可能是 I/O 等待。

**32. 索引为什么能提升查询？**

> 索引类似目录，可以减少全表扫描。但索引也会增加写入和存储成本，所以要根据查询条件、排序和过滤字段设计。

**33. 什么是幂等？**

> 同一个请求执行一次和多次，结果应该一致。比如查询接口通常幂等，创建任务可能不幂等。上传和执行 SQL 这类操作要特别注意重复提交。

**34. 你怎么回答不会的问题？**

> 我会先说我目前理解到哪一层，再说可能的排查方向。如果是没做过的生产级问题，我不会硬编，会结合项目说如果继续完善，我会怎么验证和实现。

#### Python，4 题

**1. Python 的 GIL 是干啥的？**

> GIL 是 CPython 的全局解释器锁，保证同一时刻只有一个线程执行 Python 字节码。它让内存管理更简单，但限制 CPU 密集型多线程并行。对 ChatBI 这种 I/O 比较多的后端，异步和线程池可以提升并发等待效率；如果是 CPU 密集的数据处理，可以考虑进程池、任务队列或把计算下沉到数据库 / NumPy / pandas 底层实现。

**2. 生成器、迭代器、装饰器分别是什么？**

> 迭代器是实现 `__iter__` 和 `__next__` 的对象，可以逐个取值；生成器是用 `yield` 写出来的特殊迭代器，适合流式处理大文件；装饰器是包装函数的函数，可以在不改原函数主体的情况下增加日志、鉴权、耗时统计等能力。比如 ChatBI 上传大文件时可以用生成器分批读取，接口耗时统计可以用装饰器封装。

**3. Python 的可变变量和不可变变量怎么理解？**

> 不可变对象包括 `int`、`float`、`str`、`tuple`，修改时会生成新对象；可变对象包括 `list`、`dict`、`set`，可以原地修改。项目里处理 workspace 上下文时要注意不要把默认参数写成 `{}` 或 `[]`，否则多次请求可能共享同一个可变对象。

**4. 魔术方法、静态方法、类方法、实例方法怎么理解？**

> 魔术方法是 Python 约定方法，比如 `__init__`、`__str__`、`__iter__`，用于对象初始化、打印、迭代等行为；实例方法第一个参数是 `self`，访问对象状态；类方法用 `@classmethod`，第一个参数是 `cls`，适合工厂方法；静态方法用 `@staticmethod`，不依赖实例和类状态，适合工具函数。比如 SQL 校验器可以设计成无状态静态方法，workspace repository 更适合实例方法或类方法。

#### 知识库 / RAG，15 题

**1. RAG 的发展历史简述。**

> 早期问答主要靠关键词检索和规则模板；后来出现 BM25 + 向量检索，把语义相似内容召回给模型；再往后发展出混合检索、Rerank、HyDE、Query Rewrite、多路召回、GraphRAG 和 Agentic RAG。ChatBI 当前是 RAG-lite，只做本地指标字典增强，还不是完整向量 RAG。

**2. RAG 分块策略你用的是什么，为什么？**

> 当前项目没有做长文档向量库分块，RAG-lite 用的是指标字典条目，每条指标就是一个小知识单元。这样适合 GMV、ROI、转化率这种短口径解释。如果升级到完整文档 RAG，我会按标题、段落和语义边界分块，控制 chunk 大小和 overlap，避免把一个业务定义切碎。

**3. 为什么要做这个 RAG，了解 Elasticsearch 吗？**

> 做 RAG 是为了让回答基于业务口径，而不是让模型凭记忆解释指标。Elasticsearch 更偏倒排索引和 BM25，适合关键词检索、过滤和结构化条件；向量库适合语义相似召回。真实企业知识库里常用 ES + 向量检索混合召回。

**4. RAG 的 embedding 模型用哪个，为什么？**

> ChatBI 当前没有接 embedding 模型，所以不能说用了某个 embedding。若升级，我会优先考虑中文和中英混合表现稳定的模型，比如 BGE 系列或企业内部模型，并根据召回率、延迟、成本和部署方式做评估。

**5. 为什么选择小模型作为 RAG 调用模型？**

> 当前项目没有专门部署小模型。一般 RAG 链路里可以用小模型做 query rewrite、分类、简单摘要，用大模型做复杂推理和最终回答。这样能降低成本和延迟，但关键是评测质量，不能只看模型大小。

**6. 你的 NL2SQL 是怎么实现的？**

> 当前是 Agent 调工具的工程实现，不是训练了一个 NL2SQL 模型。流程是：用户问题带 workspace 上下文，模型生成或规划 SQL，执行前经过只读 SQL、字段白名单、workspace 范围和 `SELECT *` 校验，通过后再查询数据库，最后把 SQL 和结果放进 evidence。

**7. 有没有做到 SQL 和向量数据库双联动？**

> 当前没有接向量数据库，所以不能说做了 SQL + 向量库双联动。现在是 SQL 查询 + 本地 JSON 指标字典。下一步可以做：先向量召回指标文档和表字段说明，再结合结构化 SQL 查询业务数据，最后统一给生成层。

**8. RAG 评测怎么做，指标有哪些？**

> 当前项目主要评测 RAG-lite 命中是否正确，比如问题包含 ROI 能否命中 ROI 口径。完整 RAG 可以评估召回率、Precision@K、MRR、NDCG、答案相关性、忠实度、引用覆盖率和幻觉率。

**9. PostgreSQL 和 MySQL 掌握到什么程度？**

> 我能做基础表设计、增删改查、索引、事务、连接查询和 SQL 排查。ChatBI 里更偏 PostgreSQL，用来保存业务数据、workspace 元数据和查询结果。MySQL 和 PostgreSQL 都是关系型数据库，差异包括语法细节、JSON 支持、扩展能力和事务实现等。

**10. 数据库种类有哪些？向量数据库怎么选？**

> 关系型数据库包括 MySQL、PostgreSQL；非关系型包括 MongoDB、Redis；向量数据库包括 Milvus、FAISS、ChromaDB 等。Milvus 适合服务化、大规模向量检索；FAISS 适合本地或嵌入式实验；ChromaDB 适合轻量原型。索引上常见 IVF、HNSW，距离度量常见 L2、Inner Product、Cosine。

**11. 为了解决大模型幻觉，你用了哪些方法？**

> ChatBI 里主要是五层：workspace 限制数据范围，SQL 执行前安全校验，RAG-lite 注入指标口径，回答返回 evidence，查不到的数据要求不编造。核心思路是让模型基于工具结果和证据回答。

**12. 图片召回有没有经验？**

> 这个项目没有做图片召回。我的理解是图片召回一般用 CLIP 或多模态 embedding，把图片和文本映射到同一向量空间，再做相似度检索；如果是图表图片，还可以先 OCR 或结构化解析，再结合文本召回。

**13. LangChain 组件了解多少？**

> 了解常见组件：Model、PromptTemplate、OutputParser、Retriever、VectorStore、Tool、Agent、Memory、Callback。ChatBI 更核心使用 LangGraph 做流程编排，用 MCP 做工具协议，没有重度依赖传统 LangChain Chain。

**14. 压测和 QPS 怎么做？**

> 当前项目没有做正式压测，不能报 QPS。要做的话会用 Locust、JMeter 或 wrk 分别压聊天接口、上传接口和评测接口，拆分后端接口耗时、模型耗时、MCP 工具耗时、SQL 查询耗时，并观察错误率、P95/P99 延迟和资源占用。

**15. RAG / ChatBI 的难点和痛点是什么？**

> 难点不是“能回答”，而是召回是否准、证据是否够、SQL 是否安全、上下文是否超长、工具失败怎么降级、回答怎么追溯。ChatBI 当前重点解决了数据边界、SQL 安全和 evidence，完整企业级 RAG 还需要权限、向量库、重排和系统评测。

#### Agent，15 题

**1. Agent 这块用的什么框架？**

> 项目用 LangGraph 做 Agent 流程编排，用 MCP 接外部工具。当前是单 Agent 多工具，不是完整多 Agent 生产系统。

**2. 谈谈你对 Agent 的理解。**

> Agent 不是单次问答，而是模型根据目标、上下文和工具反馈，规划下一步动作、调用工具、观察结果并继续决策的过程。工程上重点是状态、工具、边界、日志和失败恢复。

**3. LangGraph 和 AutoGen 了解吗？**

> LangGraph 更偏有状态工作流和图编排，适合明确节点、边和状态流转；AutoGen 更强调多角色 Agent 对话协作。ChatBI 选择 LangGraph，是因为当前更需要可控流程、工具调用和状态追踪。

**4. LangGraph 的边和节点怎么设置？**

> 节点可以按意图识别、工具规划、工具执行、结果汇总拆分；边根据状态决定下一步，比如普通聊天直接回答，数据问题进入 SQL / Python 工具，失败则返回错误或降级回答。当前项目的思路是先保证单 Agent 多工具闭环。

**5. 怎么解决 Agent 陷入死循环？**

> 设置最大迭代次数、工具调用次数、超时时间和停止条件；对重复调用同一工具、连续失败、无新信息返回做中断；同时记录 trace，方便发现循环原因。

**6. 多 Agent 相比单 Agent 的缺点是什么？**

> 多 Agent 会增加状态传递、任务拆分、上下文同步、成本、延迟和调试复杂度。如果任务边界不清，多个 Agent 还可能互相甩锅或重复调用工具。

**7. 懂不懂强化学习驱动 Agent？**

> 了解基本思路：用奖励信号优化 Agent 的工具选择、规划或动作策略，比如成功完成任务给正反馈。但 ChatBI 当前没有做 RL，主要是规则、提示词、工具边界和评测驱动。

**8. Function Calling 和 MCP 的区别？**

> Function Calling 更像模型直接调用开发者定义的函数，接入快；MCP 更像把工具服务标准化，适合多工具、多服务和跨环境调用。MCP 边界更清楚，但部署和排查更复杂。

**9. 有没有用过 Skills，说说原理。**

> 项目没有依赖 Skills。我的理解是 Skills 把某类任务的流程、约束、脚本和资源封装成可复用能力，模型在需要时加载对应说明和工具，相当于给 Agent 增加可复用的专业工作流。

**10. Superpower 的组件有哪些？**

> 项目里没用 Superpower。了解层面，它更像 Claude 的技能 / 工作流插件集合，常见能力围绕需求澄清、TDD、调试、代码审查、技能编写等。面试里我会说我理解技能化工作流的思路，但不把它写成项目依赖。

**11. Claude Code 常用快捷键 / 命令？**

> 这个不是项目核心能力。可以简单说我知道这类工具常用命令包括查看帮助、清上下文、压缩上下文、切换模型、查看状态等；但面试重点还是代码和工程能力，不会把快捷键当技术亮点。

**12. OpenClaw / Hermes Agent 的记忆和心跳机制了解吗？**

> 了解概念但项目没用。OpenClaw 的 heartbeat 可以按周期唤醒 Agent 执行后台任务；Hermes Agent 强调分层记忆，比如长期事实、可复用技能和会话检索。我的理解是心跳增强主动性，记忆增强连续性，但也会带来权限、成本和记忆污染风险。

**13. 谈谈你对记忆机制的认识。**

> 记忆可以分短期、长期和情景记忆。短期记忆保存当前对话上下文，长期记忆保存稳定偏好和项目事实，情景记忆保存历史任务摘要。工程上要做权限、过期、来源、可删除和防污染。

**14. 多轮对话怎么实现？**

> 基础做法是前端保存消息列表，后端把历史消息或摘要传给模型；更工程化的做法是把会话状态、workspaceId、工具结果和摘要持久化，必要时做上下文压缩，避免无限增长。

**15. Agent 怎么解决幻觉问题？**

> 和 RAG 类似，核心是让 Agent 受约束：只能基于 workspace、工具返回和 evidence 回答；工具失败时明确失败；SQL 先校验；无法确认时不要编造。

#### 微调，5 题

**1. 微调数据集怎么来？**

> ChatBI 当前没有做微调。若要做，可以从历史问答、人工标注的 NL2SQL 对、工具调用轨迹、错误修正样例和业务指标解释中构造数据，并做脱敏和质量筛选。

**2. 微调参数怎么设置？**

> 没在本项目实际训练，所以不硬报。一般会关注 base model、学习率、batch size、epoch、LoRA rank、alpha、dropout、最大长度和验证集指标，先小规模实验再扩大。

**3. LoRA / QLoRA 原理是什么？**

> LoRA 冻结原模型大部分参数，只在部分线性层旁边加低秩矩阵，训练新增的小参数；QLoRA 在量化基础模型的基础上训练 LoRA，进一步降低显存需求。它改变的是适配器参数，不是全量更新所有模型参数。

**4. 微调难吗，怎么部署，用什么卡？**

> 难点在数据质量、显存、训练稳定性和评测。ChatBI 当前没做微调部署，不能编具体机器。一般小模型 LoRA / QLoRA 可以用单卡做实验，模型越大越依赖更高显存、多卡或量化推理。

**5. 微调时 loss function 怎么计算？**

> 常见 SFT 是 next-token prediction 的交叉熵损失。模型预测下一个 token 的概率分布，和真实 token 做 cross entropy，训练目标是降低真实答案 token 的负对数似然。

#### 深度学习 / 机器学习，5 题

**1. SVM 的核函数有哪些？**

> 常见有线性核、多项式核、RBF 高斯核、Sigmoid 核。线性核适合线性可分或高维稀疏特征，RBF 更适合非线性边界。

**2. 复述 Transformer、Softmax、Top-K、Top-P、Temperature、残差、层归一化、RoPE。**

> Transformer 核心是自注意力 + 前馈网络，通过 Q/K/V 计算 token 间关联；Softmax 把 logits 转成概率；Top-K 只在概率最高的 K 个 token 里采样；Top-P 取累计概率达到 P 的候选集合；Temperature 控制随机性；残差连接帮助梯度传播；LayerNorm 稳定训练；RoPE 用旋转位置编码把相对位置信息融入注意力。

**3. 前向传播、反向传播、梯度消失爆炸怎么理解？**

> 前向传播是输入经过网络得到预测和 loss；反向传播根据 loss 对参数求梯度并更新。梯度消失是梯度越来越小，模型学不动；梯度爆炸是梯度过大，训练不稳定。常见解决方法包括残差连接、归一化、合适初始化、梯度裁剪和优化器调整。

**4. BERT 和 GPT 怎么理解？**

> BERT 是 Encoder-only，擅长理解类任务，比如分类、抽取、匹配；GPT 是 Decoder-only，擅长生成和指令跟随。ChatBI 的问答生成更接近 GPT 类模型，字段理解和检索排序也可以用 BERT 类模型。

**5. 实体关系抽取怎么做？**

> 可以用规则、传统序列标注、BERT 分类 / 抽取模型，或 LLM 结构化抽取。比如从招投标文本里抽公司、项目、金额、时间和中标关系；在 ChatBI 里类似地可以抽指标、维度、时间范围和过滤条件。

#### 后端，7 题

**1. 异步函数、高并发、async / await / asyncio.run、线程池、进程池怎么用？**

> `async def` 定义协程，`await` 等待异步 I/O，`asyncio.run` 启动事件循环。I/O 密集任务适合 async 或线程池，CPU 密集任务适合进程池。ChatBI 里模型调用、MCP、数据库更偏 I/O；大文件清洗和复杂计算可能需要任务队列或进程池。

**2. 线程和进程区别？**

> 进程有独立内存空间，隔离性好但切换成本高；线程共享进程内存，切换较轻但要注意线程安全。Python CPU 密集多线程受 GIL 限制，进程池更适合并行计算。

**3. 127.0.0.1、localhost、Docker 端口映射、内网穿透、路由、域名怎么理解？**

> `127.0.0.1` 是本机回环地址，`localhost` 通常解析到本机；Docker 端口映射把容器端口暴露到宿主机；内网穿透把本地服务暴露给公网；路由负责请求转发；域名通过 DNS 解析到 IP。前端 502 经常和这些网络层配置有关。

**4. Linux 系统会哪些？**

> 会基础命令，如 `ls`、`cd`、`grep/rg`、`ps`、`top`、`tail -f`、`chmod`、`systemctl`，能看日志、查端口、启动服务、排查进程和部署 Python/Node 项目。

**5. Redis 存储短期记忆怎么做？**

> 可以用 Redis 存会话最近 N 轮消息、摘要、工具结果或临时状态，并设置 TTL。好处是读写快，适合短期上下文；长期记忆更适合数据库或向量库。

**6. 分布式锁对多人同时操作文件有什么作用？**

> 分布式锁保证同一时间只有一个任务能修改共享资源，避免多人同时写文件导致覆盖、损坏或状态不一致。可以用 Redis `SET NX EX` 实现，并注意超时和释放锁。

**7. FastAPI 常见问题怎么答？**

> 常问路由、依赖注入、中间件、Pydantic 校验、异常处理、异步接口、CORS、文件上传和 OpenAPI 文档。结合 ChatBI 可以说：上传接口处理文件，聊天接口处理 message/workspaceId，异常处理统一返回错误，CORS 支持前端访问。

## 4. 当前实现 vs 下一版 Multi-Agent

### 4.1 当前真实实现

当前代码更准确地说是：

```text
LangGraph 主流程
  -> 意图分流节点
  -> SQL / Python / workspace / 分析工具
  -> data_analysis_agent 统一调度工具
  -> trace/evidence 返回执行过程和依据
```

这不是四个独立 Agent 文件或四个独立子图，而是一个稳定的单 Agent 多工具闭环。

### 4.2 为什么当前先这样做

先做单 Agent 多工具是为了降低第一版复杂度：

- 先保证上传、清洗、问数、报告和审计链路能跑通。
- 先把 workspace 数据边界和 SQL 安全做好。
- 先验证 MCP 工具调用能稳定工作。
- 先用 trace/evidence 解决 Agent 黑盒问题。
- 后续再拆成多 Agent 时，有清晰的业务模块和评测样例可复用。

### 4.3 下一版 Supervisor 架构

下一版可以改成：

```text
用户问题
  -> Supervisor Agent
      -> SQL Agent
      -> 数据处理 Agent
      -> 分析 Agent
      -> 问答 Agent
  -> trace/evidence
  -> 最终回答
```

### 4.4 四个子 Agent 分工

**Supervisor Agent**

- 理解用户问题。
- 判断是否需要查数、清洗、分析或直接回答。
- 拆解任务步骤。
- 选择调用哪些子 Agent。
- 汇总子 Agent 输出。
- 控制整体 trace。

**SQL Agent**

- 根据 workspace 字段和业务问题生成 SQL。
- 调用 SQL 安全校验。
- 只访问当前 workspace 授权表和字段。
- 返回结构化查询结果。
- 把 SQL、放行状态和拦截原因写入 evidence。

**数据处理 Agent**

- 对 SQL 查询结果做清洗、聚合、排序和格式转换。
- 处理缺失值、异常值和时间字段。
- 把原始数据转成适合分析的中间数据。
- 输出结构化结果给分析 Agent。

**分析 Agent**

- 做趋势、对比、异常识别和归因假设。
- 结合指标口径解释业务含义。
- 生成结论、风险和行动建议。
- 明确结论的限制和置信程度。

**问答 Agent**

- 把 SQL 结果、处理结果、分析结论和 evidence 转成自然语言回答。
- 保持回答简洁、业务化、可复述。
- 引用字段、SQL 摘要和指标口径。
- 避免编造没有数据支撑的结论。

### 4.5 面试时怎么说最稳

推荐说法：

> 我最初的设计目标是 Multi-Agent 协作，但第一版没有一上来拆成四个 Agent，而是先实现稳定的单 Agent 多工具闭环。当前代码已经有 LangGraph 编排、MCP 工具、workspace 上下文、SQL 安全、trace/evidence 和规则评测，这些都是拆成 Supervisor 四 Agent 的基础。下一版我会把 SQL、数据处理、业务分析和最终问答拆成独立子 Agent，由 Supervisor 统一编排。

## 6. 高频面试题与回答

### 6.1 这个项目是做什么的？

**短答版：**

这是一个 ChatBI 数据分析助手，用户上传业务数据后，可以用自然语言完成数据查询、图表分析和诊断报告生成。项目重点是把大模型 Agent 接入真实数据、工具调用、SQL 安全和回答依据，而不是普通聊天。

**展开版：**

这个项目面向企业经营分析场景。业务人员通常有 Excel 或 CSV，但不会写 SQL，也不一定会做复杂分析。系统会先对上传数据做清洗和字段识别，再生成业务空间。用户在某个业务空间下提问时，Agent 会基于当前 workspace 的字段、报告摘要和数据访问边界调用工具完成分析。为了保证可信，我还做了 SQL 安全、审计、Agent trace、回答依据和规则评测。

### 6.2 为什么最初想做 Multi-Agent？

**短答版：**

因为数据分析任务天然可以拆分：有人负责查数，有人负责处理数据，有人负责分析，有人负责把结论讲清楚。拆成多个 Agent 后，每个 Agent 职责更清晰，也更容易评测和维护。

**展开版：**

最初想做 Multi-Agent，是因为 ChatBI 不只是问答。一个完整数据分析任务通常包括理解问题、生成 SQL、执行安全校验、处理查询结果、做业务分析、组织最终回答。如果全部交给一个 Agent，提示词会越来越复杂，工具边界也会模糊。Supervisor + 多 Agent 的好处是把复杂任务拆开：Supervisor 负责调度，SQL Agent 负责查数，数据处理 Agent 负责整理数据，分析 Agent 负责业务判断，问答 Agent 负责最终表达。这样结构更清楚，也更容易定位问题。

### 6.3 当前为什么先做成单 Agent 多工具？

**短答版：**

因为第一阶段最重要的是把 ChatBI 主链路跑通，包括上传、清洗、workspace、SQL 安全、问数和回答依据。先做单 Agent 多工具能降低复杂度，后续再拆成 Supervisor 四 Agent 更稳。

**展开版：**

我没有一开始就拆四个 Agent，是因为多 Agent 会引入状态传递、任务协议、错误恢复和调度策略等额外复杂度。如果底层数据链路还不稳定，直接做多 Agent 反而不好调试。所以我先做了可运行的单 Agent 多工具版本，把 workspace 上下文、MCP 工具、SQL 安全、trace/evidence 和评测集先打通。这样下一版拆 Agent 时，不是在空中设计，而是基于已经稳定的模块拆分。

### 6.4 如果面试官问“代码里四个 Agent 在哪”，怎么回答？

**短答版：**

我会诚实说明：当前版本还没有拆成四个独立 Agent，代码实现是 LangGraph 单 Agent 多工具闭环。四 Agent 是我下一版 Supervisor 架构设计，当前代码已经具备拆分基础。

**展开版：**

我不会把当前代码说成四个独立 Agent 已经落地。当前代码主要是 LangGraph 主流程加 MCP 工具调用，前面有意图分流，后面由 data analysis agent 统一调用 SQL、Python、workspace 和分析工具。我的最初目标是 Multi-Agent，所以在设计时已经把能力边界拆出来了，比如 SQL 安全、数据处理、业务分析、问答输出、trace/evidence。下一版会把这些能力进一步拆成 Supervisor 编排下的四个子 Agent。

### 6.5 Supervisor 四 Agent 下一版怎么拆？

**短答版：**

Supervisor 负责拆任务和调度；SQL Agent 负责生成和校验 SQL；数据处理 Agent 负责清洗、聚合和格式转换；分析 Agent 负责趋势、异常和归因建议；问答 Agent 负责组织最终回答并引用 evidence。

**展开版：**

我会先定义统一的中间状态，比如用户问题、workspace 上下文、SQL 结果、处理后数据、分析结论和 evidence。Supervisor 读取用户问题后决定调用顺序。需要查数时调用 SQL Agent；结果需要整理时调用数据处理 Agent；需要业务判断时调用分析 Agent；最后由问答 Agent 生成自然语言回答。每个 Agent 的输入输出都结构化，这样方便记录 trace，也方便后续分别评测。

### 6.6 SQL Agent 怎么保证安全？

**短答版：**

SQL Agent 不直接执行模型生成的 SQL，而是先经过只读 SQL、workspace 表范围、字段白名单和 `SELECT *` 拦截校验，放行后才执行，并把结果写入审计和 evidence。

**展开版：**

大模型生成 SQL 不能直接信任，所以 SQL Agent 必须和安全网关绑定。首先限制 SQL 类型，只允许 `SELECT / WITH / EXPLAIN` 这类只读查询。其次，workspace 模式下只允许访问当前业务空间绑定的表。第三，字段也要校验，只能查询字段画像里存在的字段，并拦截 `SELECT *`。最后，每次放行或拦截都记录原因，返回到 trace/evidence 里。这样即使模型生成了危险 SQL，也不会直接执行。

### 6.7 数据处理 Agent 和分析 Agent 有什么区别？

**短答版：**

数据处理 Agent 负责把数据整理成可分析的结构，比如清洗、聚合、排序、缺失值处理；分析 Agent 负责解释数据，比如趋势、异常、对比、归因假设和行动建议。

**展开版：**

这两个 Agent 的边界很重要。数据处理 Agent 更偏工程和数据转换，它不负责下业务结论，而是把 SQL 返回结果处理成干净、结构化、可分析的数据。分析 Agent 更偏业务洞察，它基于处理后的数据判断指标变化、异常波动、维度贡献和可能原因。这样拆分后，一个 Agent 不会同时承担数据清洗和业务解释两种职责，输出也更容易评测。

### 6.8 问答 Agent 如何避免胡说？

**短答版：**

问答 Agent 必须基于 SQL 结果、分析结论和 evidence 回答；没有数据支撑时要说明限制，不能编造结论。

**展开版：**

我会让问答 Agent 的输入只包含结构化结果、分析结论、workspace 信息和 evidence。回答时要求引用字段、SQL 摘要或指标口径。如果没有查到数据，或者 workspace 没有入库，就明确告诉用户当前只能基于字段画像或报告摘要回答。这样可以把回答约束在已有证据范围内，减少模型凭空总结。

### 6.9 RAG-lite 和完整 RAG 有什么区别？

**短答版：**

RAG-lite 是本地指标字典关键词检索，用来补充业务口径；完整 RAG 通常会接 embedding、向量库、权限过滤和引用片段管理。当前项目实现的是 RAG-lite。

**展开版：**

我当前没有把它包装成复杂知识库。项目里是一个本地指标字典，比如 GMV、销售额、ROI、转化率等。后端根据用户问题和 workspace 字段做关键词匹配，把命中的口径注入 Agent 上下文，并返回到 evidence。它的价值是轻量、可控、容易演示。后续如果做企业级知识库，可以升级到向量检索、文档切片、权限过滤和引用追踪。

### 6.10 Text-to-SQL 评测为什么只说规则层？

**短答版：**

因为当前评测没有调用真实 LLM 生成 SQL，而是用固定样例验证 SQL 安全和 workspace 边界，所以准确说是规则层评测。

**展开版：**

我不把它说成模型准确率评测。当前评测集里每个 case 已经给定 SQL，后端调用只读 SQL 校验和 workspace 范围校验，判断危险 SQL、跨空间表、字段越权、`SELECT *` 和未入库查询能不能被拦截。这对 AI 数据应用很重要，因为这些是底线安全能力。下一步如果要评测模型效果，可以再加入自然语言问题、期望 SQL、模型生成 SQL 和结果一致性对比。

### 6.11 Agent trace 和 evidence 是怎么做的？

**短答版：**

聊天接口保留 `message`，新增 `trace` 和 `evidence`。trace 记录工具调用过程，evidence 记录 workspace、字段、SQL 和知识命中。

**展开版：**

后端在 LangGraph 执行过程中读取 `AIMessage.tool_calls` 和 `ToolMessage`，提取工具名、参数、SQL、工具结果摘要和拦截原因。最终把这些步骤组装成 trace。evidence 则从 workspace 上下文、SQL step 和 RAG-lite 命中内容里提取字段、SQL 和指标口径。前端在回答下方用折叠面板展示“执行链路”和“回答依据”。这样用户不只看到结果，也能看到结果怎么来的。

### 6.12 这个项目最大的难点是什么？

**短答版：**

最大的难点是让 Agent 基于真实业务数据回答，同时保证它不能乱查、不能越权、不能黑盒输出。

**展开版：**

难点有三类。第一是上下文，模型必须知道当前 workspace 有什么字段、什么业务类型、能查哪张表。第二是安全，大模型生成 SQL 不能直接执行，所以要加只读、表范围、字段白名单和审计。第三是可信，用户需要知道回答依据，所以要返回 trace 和 evidence。这个项目的价值就在于把数据链路、安全边界和回答依据都接进了 AI 应用。

### 6.13 如果让你继续优化，会先做什么？

**短答版：**

我会先做 Supervisor 四 Agent 重构，其次补真实 Text-to-SQL 效果评测、权限体系和向量化知识库。

**展开版：**

第一步是把当前单 Agent 多工具拆成 Supervisor + 四个子 Agent，明确每个 Agent 的输入输出和错误处理。第二步是把 Text-to-SQL 从规则层评测扩展到模型生成效果评测。第三步是补权限体系，比如用户、角色、workspace 授权和字段脱敏。第四步是把 RAG-lite 升级成权限化企业知识库。这样项目会更接近真实 AI 数据分析产品。

## 10. 量化指标怎么讲

### 10.1 如果面试官问“提效多少”

**短答版：**

这个项目目前是原型和本地演示，没有真实业务用户实验，所以我不会编“提升 80%”这种收益数字。我会按提效、降本、增速讲：提效是把 5 个手工分析动作收敛成 1 条 ChatBI 闭环，并把回答从单一 message 变成 message、trace、evidence 3 段式输出；降本是 RAG-lite 做到 0 外部向量库和 embedding 服务依赖；增速是新增指标补 1 条 JSON 记录，新工具按 1 个 MCP server/tool 接入，新安全规则补 1 条 fixture 回归。

**展开版：**

如果要讲提效，我会说它把原来需要业务人员手动完成的几个分散步骤标准化了：先整理 Excel，再判断字段含义，再写 SQL，再做图表，再整理分析结论。现在系统把这些动作串成一条 ChatBI 流程，用户上传数据后可以在 workspace 里直接问数，并看到 trace 和 evidence。  
降本方面，我没有一开始引入向量数据库、embedding 服务或外部知识库，而是先用 8 条本地指标口径做 RAG-lite，做到 Demo 阶段 0 外部知识增强依赖；同时用 5 个 MCP 服务拆开数据库、Python 图表、机器学习、业务分流和业务空间能力，降低主流程维护成本。增速方面，新指标只需要增加 1 条 JSON 字典记录，新工具按 1 个 MCP server/tool 接入，新安全风险补 1 条 Text-to-SQL fixture 就能回归。简历里还可以写 3 类上传格式、4 类图表、9 条评测样例、13 个测试文件 / 73 个测试函数，这些数字面试官追问时都能对应到代码或测试。

### 10.2 可写在简历里的量化句子

- “将 Excel 整理、字段判断、SQL 查询、图表分析、结论输出 5 个分散动作收敛为 1 条 ChatBI 闭环，支持上传后直接在 workspace 内问数和追溯依据。”
- “将聊天结果从单一 `message` 扩展为 `message + trace + evidence` 3 段式输出，展示意图、工具、SQL、字段和指标口径来源。”

- “新增指标口径只需补 1 条 JSON 字典记录，新分析工具可按 1 个 MCP server/tool 接入，降低后续能力扩展对 Agent 主流程的改动。”

### 10.3 最终可复制效果版

**适合简历放 3 条：**

- 提效：将 Excel 整理、字段判断、SQL 查询、图表分析、结论输出 5 个分散动作收敛为 1 条 ChatBI 闭环，支持 `CSV / XLSX / XLS` 3 类数据接入和 4 类常见 BI 图表展示。

**适合面试展开：**

### 10.4 提效 / 降本 / 增速拆开讲

**提效：**

- 从手工整理 Excel、判断字段、写 SQL、做图表、写结论，收敛为上传数据后在 workspace 内自然语言问数。

- Text-to-SQL 规则评测提前覆盖 6 类高风险 SQL 场景，降低越权查询、危险 SQL 和回归问题的排查成本。

**增速：**

- 新增工具能力可以按 1 个 MCP server/tool 接入。
- 新增安全规则可以补充 1 条 Text-to-SQL fixture，形成可回归样例。

## 12. 最后总结

这个项目最适合包装成“从可运行 ChatBI 原型向 Multi-Agent 数据分析系统演进”的 AI 应用项目。

最稳的主线是：

```text
真实业务数据
  -> 数据上传清洗
  -> workspace 隔离
  -> LangGraph + MCP 工具调用
  -> SQL 安全
  -> trace/evidence
  -> RAG-lite 口径增强
  -> Text-to-SQL 规则评测
  -> 下一版 Supervisor 四 Agent 演进
```

面试时要把握一个边界：

- 当前已实现：可运行的 ChatBI + 单 Agent 多工具 + 安全与可观测。
- 下一版规划：Supervisor + SQL Agent + 数据处理 Agent + 分析 Agent + 问答 Agent。

这样讲既有真实代码支撑，也能体现你对 AI 应用架构演进的理解。
